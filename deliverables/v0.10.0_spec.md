## 背景

v0.9.0 STABLE 自封"核心功能已收敛"，但**核心问题未解决**：

| 真问题 | 现状（v0.9.0）| 风险 |

| 没有公网访问入口 | localhost:8000 only | 主人不在家看不了 |

**v0.10.0 只解决 scheduler 观测部分**（其他靠 v0.12 Deployment Infra）。

## 设计

### 1. 数据持久化：JSON 文件

**位置**: `data/sync_status.json`（项目根，与 DB 同目录）

**字段**:
```json
{
  "last_success_at": "2026-06-17T03:45:12+00:00",
  "last_failure_at": "2026-06-17T03:30:01+00:00",
  "last_error": "worldcup26.ir timeout 20s",
  "last_result": {"teams": 48, "stadiums": 16, "matches": 104, "standings": 48, "synced_at": "..."},
  "consecutive_failures": 0,
  "total_successes": 42,
  "total_failures": 1
}
```

**为什么 JSON 不用 DB 表**：
- 单条记录 / 单进程 / 单 writer → 杀鸡用牛刀
- DB 表需要 alembic 迁移、ORM 定义、查询路径 → 复杂
- JSON 文件 + Lock 简单够用，原子 rename 防半写

### 2. 调用方集成

**`app/services/worldcup26_sync.py:full_sync()`**:
- 成功 → `record_success(result)` (含 teams/stadiums/matches/standings 计数)
- 失败 → `record_failure(str(exc))` 后 re-raise
- 调用方（scheduler / admin / lifespan startup）**0 改动**（自动捕获）

### 3. 读取方

| 端点 | 用途 | 鉴权 |
|---|---|---|
| `GET /health` | 健康检查 | 公开 |
| `GET /api/health/sync-status` | 数据新鲜度详情 | 公开 |

**`/health` 增强字段**：
```json
{
  "status": "healthy" | "degraded" | "unhealthy" | "unknown",
  "version": "0.9.0",
  "sync_status": { ... 上面 6 字段 ... },
  "db_row_counts": {matches, teams, standings, prediction_log, odds_snapshots},
  "scheduler_running": true
}
```

**`/api/health/sync-status` 字段**：
```json
{
  "last_success_at": "...",
  "last_failure_at": "...",
  "last_error": null,
  "last_result": {...},
  "consecutive_failures": 0,
  "total_successes": 42,
  "total_failures": 1,
  "age_seconds": 1234,           // 派生
  "freshness": "fresh"            // 派生: <30min fresh / 30-60min stale / >60min critical
}
```

### 4. 阈值（freshness）

| 条件 | freshness |
|---|---|
| age < 30 min **且** consecutive_failures < 3 | fresh |
| 30 min <= age < 60 min | stale |
| age >= 60 min **或** consecutive_failures >= 3 | critical |

### 5. Cockpit widget

位置：`#/`(Cockpit) KPI 卡片正下方

**显示**：
```

┌──────────────┬──────────────┬──────────────┬──────────────┐
│ 同步状态      │ 最近成功     │ 连续失败      │ 历史成功率    │

```

颜色编码：fresh=绿 / stale=琥珀 / critical=红 / unknown=灰

## 不做什么

- ❌ 公网部署（v0.12 Deployment Infra + 主人买服务器）
- ❌ Slack/邮件告警（v0.10 只暴露数据,告警由主人外部脚本 polling /health 触发）
- ❌ Prometheus metrics（口径不同,不做）
- ❌ HTTPS / 域名（运维范畴）
- ❌ 用户层（v0.13 - 不做）

## 预期效果

**主人能 1 秒判断**：
1. 平台是否活着（`/health` 200）
2. 数据是否新鲜（freshness field）
3. 同步是否在跑（scheduler_running + age_seconds）
4. 历史是否有失败（last_error + consecutive_failures）
