# v0.13.0 Spec — Fly.io 公网一键部署

> 2026-06-17
> **目标：让平台拥有免费的公网入口、持久卷和可重复的云端部署流程，解决主人 3 个真问题之 #1 的最后一公里。**

## 1. 背景

v0.12.0 提供 Docker + VPS 自托管方案，但主人仍需：

- 自己买 VPS、配域名、搞 HTTPS
- 手动管理 SQLite 持久化
- 自己处理 secrets

Fly.io 免费 tier 提供：

- 共享 CPU 256MB 实例 × 3（足够单实例 web）
- 1GB 持久卷
- 自动 HTTPS + 全球 Anycast
- `flyctl` CLI 一键部署

## 2. 设计目标

| 目标 | 验收标准 |
|---|---|
| 公网入口 | `https://wc2026-fifa-platform.fly.dev/health` 返回 200 |
| 一键部署 | `./deploy_fly.sh` 在设置 `FLY_API_TOKEN` 后完成全部流程 |
| 数据持久化 | 1GB 卷挂载到 `/data`，SQLite + sync_status.json 不丢 |
| Secrets 安全 | `ADMIN_TOKEN` 通过 `flyctl secrets set` 注入，不落地代码 |
| 数据迁移 | `./migrate_data_to_fly.sh` 支持把本地 DB 上传到 Fly |
| 版本动态 | `/health` 从 git tag 读取版本，防止硬编码漂移 |

## 3. 新增文件

| 文件 | 作用 |
|---|---|
| `fly.toml` | Fly.io app 配置：香港区、/data 挂载、HTTP/TCP 健康检查、rolling 部署 |
| `deploy_fly.sh` | 6 步部署：检查 → 创建 app/卷 → deploy → 健康检查 |
| `fly_secrets_set.sh` | 交互式注入 ADMIN_TOKEN / FOOTBALL_DATA_API_KEY / WC26_BASE_URL |
| `migrate_data_to_fly.sh` | 打包本地 data/ → 主人手动 SFTP → Fly 容器内解包 → 重启 |

## 4. 关键设计决策

### 4.1 `DATA_DIR` 环境变量

Dockerfile 默认 `WORKDIR /app`，数据路径 `./data`。Fly.io 挂载卷到 `/data`，因此：

- `fly.toml` 设置 `DATA_DIR: "/data"`
- `app/config.py` 读取 `DATA_DIR` 决定 SQLite 路径
- 本地开发保持默认 `./data`，云上自动切换 `/data`

### 4.2 不在 `release_command` 跑 alembic

Fly.io 的 `release_command` 在临时机运行，不挂载持久卷。SQLite 迁移需要在有 `/data` 卷的环境下执行。因此：

- schema 由 `app/main.py` lifespan startup 中 `Base.metadata.create_all` 自动创建
- 数据迁移请在本地 `alembic upgrade head` 后通过 `migrate_data_to_fly.sh` 上传

### 4.3 scheduler 必须单实例

`fly.toml` 只定义 `web` 进程，且 uvicorn `--workers 1`。APScheduler 在 lifespan 启动，多实例会导致重复同步。

### 4.4 健康检查

- `tcp_checks`：15s 间隔，2s 超时
- `http_checks`：30s 间隔，GET `/health`，40s grace period
- 部署脚本额外做 60s `/health` 轮询

## 5. 部署流程

```bash
# 1. 设置 token
export FLY_API_TOKEN=<your-token>

# 2. 一键部署
./deploy_fly.sh

# 3. 注入 secrets
./fly_secrets_set.sh

# 4. 迁移本地数据（可选）
./migrate_data_to_fly.sh
```

## 6. 已知限制

- 免费 tier 有并发/CPU 限制，世界杯开赛日流量高峰可能需要付费升级。
- 迁移脚本最后一步需要主人手动 `flyctl sftp shell` 上传（奴才不能持有 flyctl auth）。
