# v0.7.1.1 Monte Carlo 缓存 Spec

> 目标:让 `/api/simulator/tournament` 默认走缓存,减少重复 10000 sims 计算(≈4s → <50ms)。
> 范围:仅 MC Tournament 全模拟;`/simulator/groups` 的 5000 次组内模拟保持不变(已足够快)。

---

## 1. 新增数据表 `mc_run_history`

```sql
CREATE TABLE mc_run_history (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    model VARCHAR(20) NOT NULL,              -- blend | elo | glicko2
    n_sims INTEGER NOT NULL,                  -- 100 ~ 50000
    seed INTEGER NOT NULL,                    -- 默认 42
    generated_at DATETIME NOT NULL,           -- UTC, 缓存 freshness 依据
    duration_seconds REAL NOT NULL,           -- 本次实际计算耗时

    -- 6 个轮次到达概率(JSON object: team_code -> float)
    champion_distribution TEXT NOT NULL,
    finalist_distribution TEXT NOT NULL,
    semifinalist_distribution TEXT NOT NULL,
    quarterfinalist_distribution TEXT NOT NULL,
    r16_distribution TEXT NOT NULL,
    r32_distribution TEXT NOT NULL,

    -- 小组出线概率(JSON object: group_name -> {team_code -> float})
    group_advance_probability TEXT NOT NULL,

    -- 对阵频率 Top-N(JSON array)
    top_final_matchups TEXT NOT NULL,
    top_semifinal_matchups TEXT NOT NULL,

    -- 元数据
    n_teams INTEGER NOT NULL,
    n_groups INTEGER NOT NULL,
    total_matches_per_sim INTEGER NOT NULL,

    -- 复合唯一键:同 (model, n_sims, seed) 只保留最新一条
    UNIQUE(model, n_sims, seed)
);

-- 加速"最新缓存"查询
CREATE INDEX idx_mc_run_history_lookup ON mc_run_history(model, n_sims, seed, generated_at);
```

### 字段说明
- 所有分布字段用 `JSON` 字符串存储(SQLite 无 native JSON,用 `Text` + Python `json.loads/dumps`)。
- `UNIQUE(model, n_sims, seed)` 保证同一参数组合只存一条,新结果覆盖旧结果。
- `generated_at` 是缓存 TTL 判断依据,默认 **6 小时**。

## 2. API 行为变更

### 端点
```
GET /api/simulator/tournament
  ?simulations=10000   # [100, 50000], 默认 10000
  ?model=blend         # blend | elo | glicko2, 默认 blend
  ?return_top_n=8      # top N 对阵频率
  ?seed=42             # 随机种子
  ?refresh=0           # 新增:1=强制重算,0=优先缓存(默认)
```

### 缓存命中规则
1. `refresh=1` → **跳过缓存**,直接计算,计算完成后覆盖写入 `mc_run_history`。
2. 否则,查找 `(model, n_sims, seed)` 最新一条记录。
3. 若记录存在且 `generated_at` 在 **6 小时内** → 直接返回缓存。
4. 若记录不存在或已过期 → 重新计算,写缓存,返回新结果。

### 响应格式
与 v0.7.1 完全一致,额外可选字段(供调试):
```json
{
  ...
  "cached": true,           // 是否命中缓存
  "cached_at": "2026-06-16T06:00:00+00:00",
  "cache_age_seconds": 7200
}
```
> 是否加 `cached` 字段?默认不加,避免前端逻辑变化。测试环境可注入开关显示。

## 3. 新增服务函数

### `app/services/monte_carlo.py`

```python
def load_mc_cache(
    db: Session,
    model: str,
    n_sims: int,
    seed: int,
    ttl_seconds: int = 6 * 3600,
) -> Optional[dict]:
    """查询有效缓存,命中返回 API dict,否则 None."""


def save_mc_cache(
    db: Session,
    model: str,
    n_sims: int,
    seed: int,
    result: TournamentResult,
) -> None:
    """把 TournamentResult 序列化写入 mc_run_history."""


def run_mc_with_cache(
    db: Session,
    n_sims: int,
    model: str,
    return_top_n: int,
    seed: int,
    refresh: bool = False,
) -> dict:
    """缓存优先入口:命中则返回,否则 simulate_full_tournament + 写缓存."""
```

### `app/services/scheduler.py`

新增 `_job_mc_cache_warmup` + scheduler 注册:
```python
scheduler.add_job(
    _job_mc_cache_warmup,
    trigger=IntervalTrigger(hours=6),
    args=[session_factory],
    id="mc_cache_warmup",
    name="MC 缓存预热(每 6h: blend/10000/seed=42)",
    replace_existing=True,
    max_instances=1,
    coalesce=True,
)
```

Warmup 逻辑:
- 只预热**默认参数组合**: `model="blend"`, `n_sims=10000`, `seed=42`。
- 用 `run_mc_with_cache(..., refresh=False)` 触发,若缓存有效则不重复计算。
- 目的是:比赛日数据更新后(如 worldcup26.ir 15min 同步),6h 内第一次用户访问能用上新数据。

## 6. 风险与兜底

1. **CPU 阻塞**: `simulate_full_tournament` 仍是同步 CPU 密集。缓存命中时无阻塞;未命中时仍阻塞 4s。v0.7.2 可考虑 `run_in_threadpool`。
2. **存储膨胀**: 只缓存 `(model, n_sims, seed)` 组合。用户很少改参数,实际行数 <= 几十行。
3. **数据新鲜度**: 6h TTL。若比赛结果刚更新,用户点"刷新"可立即重算。
4. **seed 固定**: 默认 seed=42 导致默认缓存结果可重现,是 feature 不是 bug。
