## 二、修复决策（4 项）

### 决策 1：B-1 撤销（不是 bug）

**原标**：`prediction.py:308 (home_form - away_form)` 当 form=None 时抛 TypeError

**真核**：实测 3 个 form=None 路径全有守卫：
- `_apply_recent_form:96-97` `if home_form is None or away_form is None: return ...`
- `_factors_breakdown:308` `(home_form - away_form) if (... is not None and ... is not None) else None`
- `_reasons:370` `if home_form is not None and away_form is not None:`

**实际测试**：CIV (form=None) vs MEX (form=3) 端到端预测 → 200 OK，无抛错。**B-1 是误判**。

**保留**：18 个边界 pytest 用例（覆盖 form None 场景）作为**长期护栏**。

### 决策 2：B-2 加注释而非改代码

**问题**：`Match.kickoff_at` 存 wc26 `local_date`（美国本地 naive），`today_matches` 用 `datetime.now().replace(hour=0)`（中国本地 naive）。**两者数值上不同**，但因 SQL naive 比较 + 当前时间窗口小，**碰巧正确**。

**修法**：`matches.py:40-57` 加 18 行 docstring 显式说明：
- DB 存什么时区
- API 切片用什么时区  
- 为什么碰巧对
- 未来 kickoff_at 改 aware 时的风险 + 修法

**不修**是因为改 `kickoff_at` 字段会触发 Alembic migration + 重写 sync 逻辑 + 重灌数据（链式影响太大），超出本轮 scope。

### 决策 3：data_sync.py 删 + app/data 移

## 六、复盘 + 教训

### 1. audit 不能"估"，要"测"

最初 audit 我标了 B-1（form None 抛 TypeError），但实测 3 个路径都有守卫，**B-1 是误判**。教训：**所有"潜在 bug"必须用最小复现跑一遍再下结论**。

### 2. "碰巧正确" 也要标

B-2 today 跨时区实测**碰巧正确**（naive 比较 + 时间窗口小），但代码没注释说明这个"巧合"。master 6 个月后看到这段代码会一脸懵。教训：**"碰巧对"比"明显错"更危险**，必须显式注释。

### 3. Playwright E2E 集成才是真 E2E

之前 scripts/ 散落 5 个 p1_*_test_*.py 都是 `python script.py` 跑（不是 `pytest`），CI 集成不便。**集成进 pytest 6 项测试 21.5s 跑完**，主人未来加测试直接 `pytest tests/e2e/` 即可。

### 4. scripts 整理 "全删 vs 归档" 选归档

我选了"归档 21 个到 _archive/ + 写 README 警告"而非"全删"。理由：
- 这些脚本**对当时 v0.1.x 阶段有价值**（debug 记录 + 截图复盘）
- v0.2.0 之后**功能已被 pytest + Playwright E2E 替代**
- 归档比删更安全（万一未来 M2.0 需要重看 emoji 修复过程，verify_emoji_fix.js 还在）

### 5. git ignore 必须跑完所有流程

我先 `git add .` → 提交了 node_modules + scripts/_archive + worldcup2026.db → 然后才发现 .gitignore 没写全 → amend 修。教训：**写完 .gitignore 后必须先 `git add .` 验一遍 ls-files，再 commit**。
