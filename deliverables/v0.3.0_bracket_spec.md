# v0.3.0 Bracket 真实数据接入 Spec

**版本**：v0.3.0  
**日期**：2026-06-15  
**范围**：后端晋级/对阵生成 + 前端 Bracket 页面真实数据渲染  
**数据来源**：SQLite 已有的小组赛结果（matches + standings）

---

## 1. 目标

小组赛结束后，平台能：
1. 根据 12 个小组真实积分榜，自动计算 32 强（前 2 + 8 个最佳第三）。
2. 按 FIFA 2026 官方对阵规则，生成 Match 73-104 的真实对阵。
3. 更新 DB 中淘汰赛占位记录的 home_team_id / away_team_id / placeholder 字段。
4. 前端 Bracket 页面直接展示真实对阵，并支持 Elo 概率预测。

---

## 2. 2026 世界杯晋级规则

### 2.1 小组赛
- 48 队分 12 组（A-L），每组 4 队单循环。
- 排名规则：积分 → 净胜球 → 进球 → 直接交锋积分 → 直接交锋净胜球 → 公平竞赛 → 抽签。
- **v0.3.0 实现前 3 项**（积分、净胜球、进球）。若出现需要直接交锋才能区分的极端同分，先记录为已知限制。

### 2.2 32 强产生
- 每组前 2 名：12 × 2 = 24 队。
- 12 个小组第 3 中成绩最好的 8 队。
- 合计 32 队。

### 2.3 最佳第三排名规则
与小组赛排名规则相同：积分 → 净胜球 → 进球。

## 3. 数据模型改动

### 3.1 已有字段复用
- `Match.home_team_id` / `Match.away_team_id`：小组赛结束后写入真实球队 ID。
- `Match.home_team_placeholder` / `Match.away_team_placeholder`：写入 "2A"、"最佳第三（A/B/C/D/F）" 等可读占位。

### 3.2 不新增表
利用现有 `matches` 表，通过 `match_number`（73-104）识别淘汰赛。

## 4. API 设计

### 4.1 新增 `GET /api/bracket`
返回完整的 32 强对阵树，包含真实球队、占位、预测概率。

```json
{
  "generated_at": "2026-06-15T15:46:00Z",
  "group_stage_finished": false,
  "rounds": {
    "r32": [
      {
        "match_number": 73,
        "stage": "R32",
        "home": { "source": "2A", "team": {...}, "placeholder": "2A" },
        "away": { "source": "2B", "team": {...}, "placeholder": "2B" },
        "prediction": { "home_win": 0.42, "draw": 0.28, "away_win": 0.30 }
      }
    ],
    "r16": [...],
    "qf": [...],
    "sf": [...],
    "third_place": {...},
    "final": {...}
  }
}
```

### 4.2 新增 `POST /api/admin/bracket/rebuild`（admin only）
手动触发 Bracket 重新计算。用于小组赛结束后立即更新，或修正。

### 4.3 修改 `GET /api/matches`（可选）
小组赛结束后，返回的淘汰赛记录应带有真实 `home_team` / `away_team`。

## 5. 算法模块

### 5.1 `app/services/bracket_logic.py`

核心函数：

```python

    """计算 12 组排名（含积分/净胜球/进球）."""

def rank_third_place_teams(standings: Dict[str, List[StandingRow]]) -> List[ThirdPlaceTeam]:
    """对 12 个小组第三统一排名，取前 8."""

def resolve_bracket_matchups(
    standings: Dict[str, List[StandingRow]],
    third_places: List[ThirdPlaceTeam]
) -> List[MatchSlot]:
    """按 Annex C 规则生成 Match 73-88 的 16 个对阵."""

def update_knockout_matches(db: Session, slots: List[MatchSlot]) -> dict:
    """把计算结果写回 matches 表（home_team_id / away_team_id / placeholder）."""
```

### 5.2 最佳第三分配策略
每个 "最佳第三" 槽位有一组候选小组（如 A/B/C/D/F）。实现时：
1. 先确定 8 个晋级的小组第三。
2. 按候选小组集合的交集，贪心分配：优先把能放入唯一槽位的第三先固定。
3. 若多个槽位竞争同一个第三，按 FIFA 官方 Annex C 的优先级处理（需要进一步查证）。

**简化版实现（v0.3.0）**：按排名从高到低，依次选择可放入的最小编号槽位。

## 6. 前端改动

### 6.1 `app/static/js/app.js`
- `renderBracket()` 改为优先调用 `/api/bracket`。
- 若 `/api/bracket` 返回真实球队，则渲染真实对阵。
- 若小组赛未结束（`group_stage_finished=false`），保留现有小组赛折叠视图 + 淘汰赛 placeholder。
- 在淘汰赛节点上叠加 Elo 预测概率条。

### 6.2 新增 `renderBracketNodeReal(match)`
渲染真实对阵节点，包含：
- 两队国旗、中文名
- 比分（若已完赛）
- Elo 胜率条
- 跳转比赛详情

## 9. 风险与限制

| 风险 | 影响 | 缓解 |
|------|------|------|
| FIFA Annex C 最佳第三分配细节未完全确认 | 对阵可能错误 | 用 worldcupper.com/predictthecup.com 交叉验证；实现可配置映射 |
| 小组赛同分需直接交锋/公平竞赛区分 | 排名可能不准确 | v0.3.0 先实现前 3 项规则，并在文档中说明限制 |
| 前端现有淘汰赛筛选逻辑与 DB stage 不一致 | 页面显示异常 | 同步修复前端筛选条件 |
| 小组赛结束前 Bracket 更新无意义 | 频繁 rebuild 浪费资源 | admin 手动触发 + scheduler 仅在小组赛阶段后启用 |

## 10. 非目标

- 不接入 StatsBomb Elo（留到 v0.4.0）。
- 不实现完整的公平竞赛/抽签 tie-breaker（超出 v0.3.0 范围）。
- 不修改小组赛赛程数据同步逻辑。
