# v0.16.2 Spec — 前端首选/次选比分展示优化

> 2026-06-24
> **目标：在已有比分推荐 v2.1 基础上，把 `primary_score` / `secondary_score` 同时展示到前端关键位置，降低用户对单一推荐比分的依赖，提升信息透明度。**

## 1. 背景

v0.16.1 在后端完成了比分推荐 v2.1：

- `recommended_score`：全局最可能比分
- `outcome_aligned_score`：与胜平负结果对齐的比分
- `primary_score`：首选比分（推荐度更高）
- `secondary_score`：次选比分（备选结果）
- `score_reliability_stars`：比分推荐度星级

但前端当时只展示了 `recommended_score`，用户看不到模型给出的前两名推荐结果。v0.16.2 把 `primary_score` / `secondary_score` 同步展示到首页焦点战、比赛详情 AI 预测区、赛后复盘三个核心位置。

## 2. 设计目标

| 目标 | 验收标准 |
|---|---|
| 首页焦点战展示首选/次选 | 首页焦点战卡片的「首选比分」格显示 `primary_score`，下方显示「次选」 |
| 比赛详情页展示首选/次选 | `/api/matches/{id}/prediction` 返回的 `primary_score` / `secondary_score` 渲染到 AI 预测区标题 |
| 赛后复盘展示首选/次选 | 已完赛比赛的复盘卡片显示预测首选/次选，并与实际比分对比 |
| 向后兼容 | 旧数据若无 `primary_score`，自动回退到 `recommended_score` |
| 缓存刷新 | Service Worker 缓存版本保持 `wc2026-v6`，静态资源 query string 升级到 `?v=0.16.2` |

## 3. 修改文件

| 文件 | 改动 |
|---|---|
| `app/main.py` | `APP_VERSION` 常量升至 `0.16.2` |
| `app/static/js/app.js` | 3 处渲染逻辑改为展示 `primary_score` / `secondary_score`，并保留 `recommended_score` 回退 |
| `app/static/index.html` | CSS/JS 静态资源 query string 改为 `?v=0.16.2`，Tab 注释更新 |
| `app/static/sw.js` | 缓存版本注释更新为 v0.16.2 |
| `README.md` | banner、文档版本、API 速查、阶段描述同步到 v0.16.2 |

## 4. 关键设计决策

### 4.1 回退策略

`primary_score` / `secondary_score` 字段在 v0.16.1 才引入，缓存或旧数据可能缺失。前端统一使用 `primary_score || recommended_score` 作为首选，`secondary_score || '—'` 作为次选，保证任何情况下都有合理展示。

### 4.2 不升 Service Worker 大版本

本次仅前端展示调整，无新增静态资源或缓存策略变化，因此保持 `CACHE_VERSION = 'wc2026-v6'`。通过升级 `index.html` 中 `app.js` 的 `?v=0.16.2` 确保浏览器重新加载 JS。

### 4.3 最小侵入

仅修改 `app.js` 中 4 处与比分展示相关的模板字符串，不改动数据获取、路由、状态管理或其他业务逻辑。
