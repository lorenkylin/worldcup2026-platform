### 2.1 football-data.co 免费层 API 实际可用端点
| 端点 | 用途 | 免 Key 可用? |
|------|------|-------------|
| `GET /v4/matches` | 比赛列表(可按日期范围) | ✅ |
| `GET /v4/teams/{id}` | 球队详情 | ✅ |
| `GET /v4/teams` | 球队列表 | ✅ |
| `GET /v4/competitions` | 赛事列表 | ✅ |
| `GET /v4/competitions/{id}/standings` | 积分榜 | ✅ |
| **赔率端点** | ❌ **免费层无赔率 API**(仅 X-Widget 引用商业 feed) | ❌ |

**结论**:
- ✅ 用 football-data 同步 **比赛元数据**(日期/状态/比分)作为 wc26 主源的交叉验证
- ❌ **赔率走势** 仍由现有 4 家公司种子数据驱动 + **自动快照机制**(即使值不变,6h 打点)

### 2.2 赔率走势数据驱动的现实约束
- 主人说"单场赔率时间序列"——需要历史快照表
- 现状: `MatchOdds` 表只有"最新一条",无历史
- 必须新增 **`odds_snapshots` 表**(每次同步/录入都追加一行)
- 走势曲线初始会"平坦"(因为赔率值未变),但有真实时间戳
- 未来接入付费赔率 API 时,无需改表结构

### 2.3 6h 刷新范围决策
| 数据类型 | 当前刷新 | 建议 |
|---------|---------|------|
| wc26.ir 主源(比赛/积分) | 15min | **保持 15min**(实时比分) |
| recent_form(B2) | 30min | **保持 30min** |
| Bracket 重算 | 15min | **保持 15min** |
| football-data.co 元数据 | 无 | **新增 6h**(交叉验证) |
| odds_snapshots(走势) | 无 | **新增 6h**(打点) |
| Hicruben/StatsBomb Elo | 手动 | **不调度**(重算耗 CPU) |

**6h 调度器的核心定位**:补充型同步,不影响实时数据。

## 三、技术方案

### 3.1 模块 A:football-data.co 客户端 + 缓存

**新增文件**:
- `app/services/football_data.py` — HTTP 客户端 + 速率限制 + 内存缓存

**设计**:
```python
class FootballDataClient:
    BASE_URL = "https://api.football-data.org/v4"
    RATE_LIMIT_PER_MIN = 10
    CACHE_TTL_SECONDS = 900  # 15min

    def __init__(self):
        self._cache: Dict[str, Tuple[float, Any]] = {}  # key → (timestamp, data)
        self._request_times: Deque[float] = deque()  # 滑动窗口

    def _wait_for_rate_limit(self):
        """确保 60s 内最多 10 个请求。"""
        now = time.monotonic()
        # 移除 60s 外的请求时间戳
        while self._request_times and now - self._request_times[0] > 60:
            self._request_times.popleft()
        if len(self._request_times) >= 10:
            sleep_time = 60 - (now - self._request_times[0])
            time.sleep(sleep_time)
        self._request_times.append(time.monotonic())

    def _cached_get(self, path: str) -> Optional[dict]:
        """先查缓存,未命中则请求 + 写缓存。"""
        if path in self._cache:
            ts, data = self._cache[path]
            if time.time() - ts < self.CACHE_TTL_SECONDS:
                return data  # 缓存命中

        self._wait_for_rate_limit()
        resp = httpx.get(f"{self.BASE_URL}{path}", headers=self._headers(), timeout=20)
        resp.raise_for_status()
        data = resp.json()
        self._cache[path] = (time.time(), data)
        return data

    def get_matches_by_date_range(self, date_from: str, date_to: str) -> list:
        """日期范围 ISO 8601: date_from=YYYY-MM-DD, date_to=YYYY-MM-DD"""
        return self._cached_get(f"/matches?dateFrom={date_from}&dateTo={date_to}")

    def get_team(self, team_id: int) -> dict:
        return self._cached_get(f"/teams/{team_id}")
```

### 3.2 模块 B:赔率快照(odds_snapshots)

**新增 Alembic 迁移** `b9c4d8e2f1a3_add_odds_snapshots_table.py`:

```python
class OddsSnapshot(Base):
    """单场比赛每家公司每个时间点的赔率快照(用于走势图表)."""

    __tablename__ = "odds_snapshots"

    id = Column(Integer, primary_key=True)
    match_id = Column(Integer, ForeignKey("matches.id"), nullable=False, index=True)
    bookmaker = Column(String(50), nullable=False, index=True)
    home_win = Column(Float, nullable=True)
    draw = Column(Float, nullable=True)
    away_win = Column(Float, nullable=True)
    over_2_5 = Column(Float, nullable=True)
    under_2_5 = Column(Float, nullable=True)
    snapshot_at = Column(DateTime, default=lambda: datetime.now(timezone.utc), index=True)
    source = Column(String(20), default="snapshot")  # snapshot/manual/api

    __table_args__ = (
        Index("ix_odds_snap_match_book_time", "match_id", "bookmaker", "snapshot_at"),
    )
```

**API**:
- `GET /api/matches/{id}/odds/history` — 返回某场比赛的所有 snapshot


### 3.4 模块 D:6h 调度器

**scheduler.py 新增任务**:

```python
# 6h 刷新:football-data.co 元数据 + odds 快照打点
scheduler.add_job(
    _job_periodic_refresh,
    trigger=IntervalTrigger(hours=6),
    args=[session_factory],
    id="periodic_6h_refresh",
    name="6h 周期刷新(fb-data 元数据 + odds 快照)",
    replace_existing=True,
    max_instances=1,
    coalesce=True,
)
```

**`_job_periodic_refresh`**:
1. 调用 `FootballDataClient.get_matches_by_date_range()`(未来 7 天 + 过去 7 天)
2. **只更新已有比赛的 status / kickoff_at / score**(不创建新比赛,避免与 wc26 冲突)
3. 给所有现有 `MatchOdds` 行追加 `OddsSnapshot`(值不变也记录)
4. 写入 `ApiUsageLog`(football_data + odds_snapshot)

**lifespan startup** 立即跑一次(避免 6h 调度窗口期数据 stale)

## 五、风险与权衡

| 风险 | 缓解 |
|------|------|
| football-data.co 缺赔率端点 | 仅作元数据交叉验证,赔率走势仍由人工/快照驱动 |
| 6h 刷新与 15min 主同步冲突 | refresh 只更新元数据,不创建/删除比赛(幂等) |
| odds_snapshots 表增长 | Alembic 加索引 (match_id, bookmaker, snapshot_at);后续可加 30 天 TTL 清理 |
| 真实 football-data.co 需免费 Key | 主人需注册 https://www.football-data.org/ 取 token,填 .env `FOOTBALL_DATA_API_KEY` |
| 免费层 10 req/min 限制 | 内存缓存 15min + 滑动窗口限速 + startup 一次性 6h 拉取 ≤20 个请求 |
| API 限速触发 429 | catch → ApiUsageLog status=throttled,下次重试 |
