## 二、详细说明

### 2.1 统一 Elo→Poisson-λ 参数

**问题**：`app/services/prediction.py` 与 `app/services/monte_carlo.py` 各自定义了 `_elo_to_lambda`，但参数不同：

| 来源 | HOME_ADVANTAGE | BASE_LAMBDA | GOAL_PER_ELO_DIFF |
|---|---|---|---|
| `prediction.py` | 60 | 1.35 | 0.0035 |
| `monte_carlo.py` | 80 | 1.3 | 0.005 |

`monte_carlo.py` 注释写“与 prediction.py 保持一致”，实际不一致。

**修复**：
- 在 `app/config.py` 的 `Settings` 中新增模型参数：
  - `poisson_home_advantage: float = 60.0`
  - `poisson_base_lambda: float = 1.35`
  - `poisson_goal_per_elo_diff: float = 0.0035`
  - `poisson_lambda_floor: float = 0.3`
- 新增 `app/services/elo_params.py`，提供统一的 `elo_to_lambda(home_elo, away_elo)`。
- `prediction.py`、`monte_carlo.py` 删除本地 `_elo_to_lambda`，改从 `elo_params` 导入。
- `simulator.py`、`backtest.py` 的导入同步调整。
- 更新 `tests/test_api.py`、`tests/test_breakdown.py`、`tests/test_prediction.py` 中硬编码断言为引用 `settings.poisson_*`。
- `.env.example` 增加对应环境变量示例。

行为保持向后兼容：默认值沿用原 `prediction.py` 的 60/1.35/0.0035/0.3。

### 2.2 修复 `mc_run_history` Alembic autogenerate 噪音

**问题**：模型 `MCRunHistory` 声明 `id = Column(Integer, primary_key=True, index=True)`，但数据库实际只有复合索引 `ix_mc_run_history_lookup(model, n_sims, seed, generated_at)`。每次 `alembic revision --autogenerate` 都会提示删除复合索引、新增 `ix_mc_run_history_id`。

**修复**：
- `app/models.py`：`id` 去掉冗余的 `index=True`（主键已隐式索引）。
- 在 `__table_args__` 显式声明复合索引：
  ```python
  Index("ix_mc_run_history_lookup", "model", "n_sims", "seed", "generated_at")
  ```
- 确保 `Index` 已从 `sqlalchemy` 导入。

未新建 Alembic 迁移，因为模型已匹配现有 schema。

### 2.3 修复 E2E 首个测试 setup 超时 flaky

**问题**：`tests/e2e/conftest.py` 的 `page` fixture 直接 `page.goto(base_url)`，第一个 E2E 测试启动时 uvicorn 可能还没就绪，30s 超时。

**修复**：
- 在 `tests/e2e/conftest.py` 增加 `_wait_for_server(base_url)`，轮询 `/health` 直至 200。
- 增加模块级 autouse fixture `_server_ready(base_url)`，确保所有 E2E 测试导航前服务已就绪。
- `page` fixture 中把默认导航超时提升到 60s。
- 仅使用 stdlib，无新增依赖。

### 2.4 彻底解决球场变体重复

**问题**：v0.13.0 数据清理后 `stadiums=16`，但服务重启同步后复发为 17，原因是：
- API 返回 `"GEHA Field at Arrowhead Stadium"`，DB 存 `"GEHA Field at Arrowhead"`。
- 早期 `_normalize_stadium_name` 同时去掉尾部 `Stadium` 和 `Field`，导致两者被分到不同 key。
- 补齐后，又发现 API 国家为 `"United States"`，DB 旧记录为 `"USA"`，`sync_stadiums` 按原值匹配失败，再次插入新记录。

**修复**：
- `app/services/worldcup26_sync.py`：
  - `_normalize_stadium_name` 只去尾部 `Stadium`，保留 `Field`。
  - 新增 `_normalize_country`，把 `USA/US/United States of America` 统一为 `united states`。
  - `sync_stadiums` 兜底匹配改为：先按城市查候选，再用规范化国家 + 规范化球场名做匹配。
- `scripts/cleanup_v013_data.py`：同步调整 `_normalize_stadium_name`，只去尾部 `Stadium`。

服务重启后验证 `stadiums=16` 稳定，且 Arrowhead 名称自动对齐为 API 返回的正式名 `"GEHA Field at Arrowhead Stadium"`。
