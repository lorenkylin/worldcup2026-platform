# v0.12.0 Spec — Deployment Infra 部署基础设施

> 2026-06-17
> **目标：让平台从 localhost 进入可公网部署、CI 自动测试、一键上线状态。**

## 1. 背景

v0.10.0 解决 scheduler 观测，v0.11.0 解决真实准确率机制，但平台仍只能在本地 `uvicorn` 运行。主人 3 个真问题之 #1 部署/可用性还差最后一块：

- 无生产镜像 / 无容器编排
- 无 CI，每次 push 靠人工跑测试
- 无标准化部署脚本，VPS 环境差异大

## 2. 设计目标

| 目标 | 验收标准 |
|---|---|
| 生产镜像 | `docker build -t wc2026 .` 成功，体积 < 250MB |
| 单实例编排 | `docker compose up -d` 拉起服务，`/health` 200 |
| 一键部署 | `./deploy.sh` 在干净 VPS 上完成 git pull → build → health check |
| CI | push/PR 自动跑 pytest + alembic 干跑 + bash 语法检查 |
| 安全 | 非 root 运行，不打包本地 `.db` |

## 3. 新增文件

| 文件 | 作用 |
|---|---|
| `Dockerfile` | python:3.13-slim-bookworm + 非 root + healthcheck + workers=1 |
| `.dockerignore` | 排除 venv/data/tests/.git，减体积 |
| `docker-compose.yml` | 单服务 + 持久卷 + env 覆盖 + healthcheck |
| `deploy.sh` | 严格模式 + 前置检查 + build + 60s 健康检查 |
| `.github/workflows/ci.yml` | pytest（非 E2E）+ alembic --sql 干跑 + import sanity |

## 4. 关键设计决策

### 4.1 Dockerfile 必须 workers=1

APScheduler 在 lifespan startup 启动 full_sync + 6h 周期任务。多 worker 会导致重复同步、写冲突。因此 `CMD` 固定 `--workers 1`。

### 4.2 数据持久化走挂载卷

SQLite + `data/sync_status.json` 通过 `docker-compose.yml` 挂载 `./data:/app/data` 与 `./logs:/app/logs`，避免镜像内丢数据。

### 4.3 CI 不跑 E2E

E2E 需 live server，CI 环境无稳定 uvicorn。CI 跑非 E2E + import sanity；E2E 由本地/预发布环境手动或单独 job 跑。

### 4.4 健康检查双重保障

- Dockerfile `HEALTHCHECK`：Docker daemon 层
- docker-compose `healthcheck`：compose 层
- deploy.sh `curl -f /health`：部署脚本层

## 5. 测试覆盖

- `tests/test_v012_deployment.py`：34 个断言覆盖文件存在性、关键字段、YAML 真解析、bash 语法、应用可 import。

## 6. 已知限制

- HTTPS / certbot 仍由主人自行处理（v0.13 Fly.io 解决公网入口）。
- 首次部署需手动 `alembic upgrade head` 或依赖 lifespan `create_all`。
